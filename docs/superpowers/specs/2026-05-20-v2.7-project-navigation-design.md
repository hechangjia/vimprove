# v2.7 Project Navigation Design

## Purpose

Vimprove now teaches core editing, Visual mode, macros/registers, marks/jumps, real-world setup, and daily substitute workflows. The next practical gap is how real Vim users move around a project: listing buffers, switching files, opening splits, and using window commands without leaving the keyboard.

v2.7 should add a small, simulated project-navigation layer that teaches the mental model without pretending the browser has real file IO.

## Recommended Scope

Build a focused v2.7 release with three connected parts:

1. **Buffer list simulation**
   - Add buffer metadata to `VimState`.
   - Support `:ls` / `:buffers`.
   - Support `:bnext` / `:bn`, `:bprevious` / `:bp`, and `:buffer {N}`.
   - Switching buffers saves the current visible buffer back into the current simulated buffer and loads the target buffer.

2. **Window split simulation**
   - Add lightweight window metadata to `VimState`.
   - Support `:split`, `:vsplit`, and `:close`.
   - Support `Ctrl-w h/j/k/l` and `:wincmd h/j/k/l` for keyboard window focus movement.
   - Keep rendering simple: the editor may still show one active buffer while status metadata communicates the simulated split state.

3. **Chapter 12: Project Navigation**
   - Teach buffers as open files, split windows as views, keyboard window focus, and a practical navigation routine.
   - Add a Window Navigator minigame to drill `Ctrl-w h/j/k/l`, covering a skill not trained by Snake, 2048, or Find Target.

## Excluded From v2.7

- Real filesystem-backed file opening, write, or save state.
- Tab pages, argument lists, netrw, fuzzy finding, quickfix/location lists, and LSP navigation.
- Full multi-pane editor rendering with independent scroll/cursor UI.
- Named buffer deletion and hidden/modified buffer semantics beyond simple status text.

## Architecture

Extend `VimState` with simulated `buffers`, `currentBufferIndex`, `windows`, `currentWindowIndex`, and `pendingCtrlW`. Keep `state.buffer` and `state.cursor` as the active visible editing surface for backwards compatibility. Ex commands should live in `src/core/exCommands.ts`; helper functions should keep buffer/window state normalized.

Lessons continue through the existing config-driven curriculum. The new minigame should use a `window-navigator` content block and a React component in `src/components/minigame`, following existing minigame patterns.

## Success Criteria

- Unit tests cover buffer listing/switching and split/window navigation commands.
- `Ctrl-w h/j/k/l` works in normal mode and clears pending state on `Escape`.
- Chapter 12 has four lessons and matching zh / zh-lively translation structure.
- The Window Navigator game renders from a lesson content block.
- `npm run typecheck`, `npm run lint`, focused tests, quickcheck, and build pass.
